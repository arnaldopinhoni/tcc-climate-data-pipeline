TCC-CLIMA - ARQUITETURA DO PROJETO

1) Objetivo
Construir um pipeline de dados climaticos orientado a camadas para coletar previsoes da Open-Meteo,
armazenar bruto (Bronze), preservar historico por ingestao nas camadas analiticas e disponibilizar
dados prontos para dashboard e analise.

2) Fluxo de dados (visao macro)
Open-Meteo API -> Ingestao Python -> Bronze (raw) -> dbt Silver History/Latest -> dbt Gold History/Latest -> Dashboard

3) Camadas e responsabilidades

3.1 API (Open-Meteo)
- Fonte externa de dados meteorologicos horarios.
- Parametros (URL, timezone, variaveis horarias, timeout) controlados por variaveis de ambiente.

3.2 Ingestao Python (ETL)
- Script principal: etl/ingest/open_meteo_ingest.py
- Le cidades de OPEN_METEO_CITIES_JSON (multi-cidade).
- Para cada cidade, chama a API e grava o payload bruto no Bronze.
- Cliente API: etl/utils/api_client.py
- Conexao DB: etl/utils/db_connection.py

3.3 Bronze (dados brutos)
- Tabela principal: public.bronze_climate_raw
- Estrutura logica:
  - id
  - city
  - raw_json
  - ingestion_time
  - created_at
- Funcao: persistir o payload bruto por cidade sem perda de detalhe, com timestamps timezone-aware.

3.4 Silver (dados horarios tratados)
- Modelos dbt:
  - dbt/models/example/silver_climate_hourly_history.sql
  - dbt/models/example/silver_climate_hourly.sql
- Silver historico:
  - expande todos os payloads do Bronze
  - preserva bronze_record_id e ingestion_time
  - gera uma serie horaria versionada por rodada de ingestao
- Silver corrente:
  - seleciona apenas a rodada mais recente por cidade
  - mantem a interface simples para consumo atual e dashboard

3.5 Gold (dados agregados para consumo)
- Modelos dbt:
  - dbt/models/example/gold_daily_summary_history.sql
  - dbt/models/example/gold_daily_summary.sql
- Gold historico:
  - agrega por city + bronze_record_id + ingestion_time + day
- Gold corrente:
  - agrega a rodada mais recente por cidade
- Metricas principais:
  - avg_temp
  - max_temp
  - total_precipitation
  - avg_dew_point_2m
  - avg_shortwave_radiation
  - avg_wind_speed_10m
  - avg_vapour_pressure_deficit
  - total_et0_fao_evapotranspiration

3.6 Qualidade de dados
- Testes dbt de not_null em city/day/record_time e nos campos de rastreabilidade ingestion_time/bronze_record_id.
- Testes de unicidade composta:
  - dbt/tests/unique_silver_city_record_time.sql
  - dbt/tests/unique_silver_history_bronze_record_time.sql

4) Orquestracao (Airflow)
- DAG: airflow/dags/full_pipeline_open_meteo.py
- Sequencia:
  1. bronze_ingest
  2. dbt_run
  3. dbt_test
- Execucao com ambiente UV local do container:
  - /opt/airflow/.venv/bin/python
  - /opt/airflow/.venv/bin/dbt
- dbt target/log direcionados para /tmp para evitar problema de permissao em volume montado.

5) Runtime e infraestrutura
- Docker Compose: docker-compose.yml
- Servicos:
  - postgres (armazenamento)
  - airflow (orquestracao)
- Bootstrap UV do Airflow:
  - Script: airflow/start_airflow.sh
  - uv sync --frozen --no-dev --no-install-project
  - sobe scheduler e webserver

6) Gerenciamento de dependencias

9) Decisoes arquiteturais (por que)
- Bronze com raw_json: preserva a fonte original para auditoria e reprocessamento.
- Silver/Gold com versao historica e versao corrente: permite rastreabilidade por ingestion_time sem perder simplicidade no consumo atual.
- Airflow + dbt: orquestracao e transformacao com responsabilidades claras.
- UV: reproducibilidade de ambiente e controle de dependencias.
- Multi-cidade por configuracao: escala sem alterar codigo da DAG.

10) Estado atual

- Silver e Gold com suporte a historico por ingestion_time e views correntes derivadas do historico.
- DAG validada com tarefas bronze_ingest, dbt_run e dbt_test.
